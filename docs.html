Chisel UI Docs

Everything you need to use, customize, and extend Chisel UI.

Version 1.0.0 · Last updated April 2026

Introduction

Chisel UI is a precision-built Figma component library designed for product designers and design system teams who care about the details. Every component is built token-first, with full auto layout, component properties, and light and dark mode included out of the box.

This library is not a shortcut — it is a foundation. The goal is not to replace your design thinking but to give you a system that handles the structural decisions so you can focus on the product.

What's included

  • 47 components across 8 categories
  • Full token system — color, typography, spacing, radius, shadow, motion
  • Light and dark mode for every component
  • Auto layout on every frame
  • Figma Variables support
  • Swap-ready icon set
  • 6 full-page templates

Who's it for

  • Product designers building digital products
  • Design system leads establishing a token-based foundation
  • Freelancers who want a professional starting point without building from scratch

Philosophy

Good design has a system. Chisel UI is built on the belief that consistency is not a constraint — it is what makes products feel considered. Every decision in this library, from the 8pt spacing scale to the five-level elevation system, exists for a reason.

Getting Started

Getting Started

2.1 Accessing the file

Option A — Figma Community

Search "Chisel UI" on the Figma Community page, or use the direct link in the navigation. Click Duplicate to add the file to your drafts.

Option B — Direct link

Click "Open in Figma" from any page on the Chisel UI website. This takes you directly to the Community file.

Once duplicated, the file lives in your Figma drafts. From there you can move it to any project or team.

2.2 File structure

The Figma file is organized into seven pages:

  • Cover — File metadata, version number, last updated date, and a quick summary of what changed in this release.
  • Foundations — The raw design system building blocks — color palette, type scale, spacing scale, radius scale, elevation system, grid specifications, and the icon library. This is the reference page. You should not need to edit anything here.
  • Components — All 47 components organized by category — Forms, Navigation, Feedback, Data Display, Layout, Overlays, Typography. Each component is presented with all its variants and states laid out in a structured frame. Component properties are set and ready to use.
  • Tokens — The full variable system. Every token is defined here as a Figma Variable with light and dark mode values. This is where you go if you want to customize the system for your brand.
  • Icons — The icon set organized into categories. All icons are components with a consistent 24px frame. Swap icons using the component panel.
  • Templates — Six full-page templates assembled from Chisel UI components — a landing page, a dashboard, a settings page, a login page, a pricing page, and a component showcase.
  • Changelog — Version history in reverse chronological order. Every addition, change, and fix is documented here.

2.3 Using as a team library

To make Chisel UI components available in all your team's Figma files:

  1. Move the duplicated file into your team workspace (not personal drafts)
  2. Open the file
  3. Click the Figma menu (top left) → Libraries
  4. Click "Publish library"
  5. Add a description and click Publish
  6. In any other file in your team, open the Libraries panel and enable Chisel UI

Once enabled, all components appear in the Assets panel under "Chisel UI" and can be dragged directly into your designs.

2.4 Requirements

info

Chisel UI uses Figma Variables for the token system. This feature is available on all Figma plans including Free. However, Variable Modes (used for switching between light and dark mode) require a paid Figma plan.

If you are on the free plan, you can still use all components — they will open in their default (light) mode. Dark mode switching via Variables will not be available, but you can still view dark mode variants on the Components page.

Minimum Figma version: December 2023 release or later (Variables support required)

Foundations

Color

The color system is built on two layers — a raw palette and a semantic token layer mapped on top of it.

Raw palette

The palette contains nine color ramps. Each ramp has seven stops from lightest (50) to darkest (900). The ramps are: Gray, Blue, Teal, Coral, Pink, Green, Amber, Red, and Purple. These are the source-of-truth values. You should never use raw palette values directly in your designs — always use the semantic tokens.

Semantic tokens

Semantic tokens give meaning to raw palette values:

color/bg/secondaryElevated surface (cards, panels)

Light and dark mode

Every semantic token has a light and dark mode value. Switching modes in Figma swaps all token values simultaneously — no manual recoloring needed. To switch a frame between light and dark mode, select the frame, open the Variables panel, and change the mode.

Customizing colors

To use your own brand colors, open the Tokens page, select the variable collection, and override the semantic token values. You do not need to touch the raw palette. See Section 7 (Theming) for a full walkthrough.

Typography

Chisel UI uses three typefaces, each with a specific role:

  • Audiowide — display only. Used for all titles, section headings, component names, stat values, and the logo. Never use Audiowide for body text or small labels — it is a display face and becomes illegible below 14px.
  • Orbitron — UI labels only. Used for all uppercase labels, badges, nav links, buttons, eyebrow text, and metadata. Always used uppercase with letter-spacing 0.08em. Available in Regular (400) and Medium (500).
  • DM Sans — body only. Used for all descriptions, body copy, list items, muted subtitles, placeholder text, and longer-form content. Available in Regular (400) and Medium (500).

Shadows & Elevation

Chisel UI uses a five-level elevation system. Higher numbers mean the element is visually closer to the user — more elevated, more important.

1elevation/1Subtle lift — badges, tags, strip icons
2elevation/2Cards at rest

Dark mode shadows

Shadows become stronger in dark mode to remain visible on dark backgrounds. The elevation tokens automatically adjust — you do not need to set separate shadow values for dark mode.

The glow

Chisel UI includes a single glow effect (--glow) using the gold accent color. This is used sparingly — only on the final CTA section and on the floating Figma pill button. Do not use the glow on every element.

Iconography

The icon set is included on the Icons page of the Figma file.

Specifications

  • Frame size: 24px × 24px
  • Style: outlined, 1.5px stroke weight
  • Corner radius on strokes: rounded
  • All icons are components — do not detach them

Sizing

Icons are designed at 24px but can be scaled to 16px or 20px while remaining legible. Do not use icons below 16px.

Using icons in components

All Chisel UI components that accept an icon use an instance swap property. Select the component, find the icon property in the component panel, and swap to any icon from the icon set.

Adding icons to your own designs

Go to the Icons page, find the icon you need, copy it (Cmd/Ctrl + C), and paste it into your file. If you have Chisel UI enabled as a team library, icons are also available in the Assets panel.

Requesting a missing icon

Open a GitHub issue with the label "icon request" and describe the icon needed. Include the use case and any reference examples.

Tokens

What are design tokens

Design tokens are named values that store design decisions. Instead of writing "the button background is #0F0E0C", you write "the button background is color/button/face". The token holds the actual value.

This matters because:

  • When you change a token value, every element using that token updates automatically
  • Tokens make it possible to switch themes (light to dark, or custom brand) by swapping the values behind the names
  • Tokens create a shared language between design and development

In Chisel UI, tokens are implemented as Figma Variables. Every color, spacing value, radius, and shadow in the system is a named token — nothing is hardcoded.

Token naming structure

All tokens follow this naming pattern:

[category] / [subcategory] / [variant]

Examples:

  • color / bg / primary
  • color / text / muted
  • color / accent / gold
  • spacing / section / padding-md
  • radius / component / card
  • elevation / level / 2
  • motion / duration / base
  • motion / easing / spring

The category is always the broadest grouping. The subcategory narrows the context. The variant specifies the exact use case.

Figma Variables

All tokens are stored as Figma Variables in a collection called "Chisel UI Tokens." The collection has two modes: Light and Dark.

Viewing variables

Open the Variables panel from the right sidebar (click the grid icon) or from the main menu → Edit → Variables.

Applying a variable to a property

Select a layer, click the property you want to apply a variable to (e.g. Fill), then click the variable icon (4 dots) next to the value field. Choose from the token list.

Switching modes

Select a frame, go to the Design panel, find the Variable Mode section, and switch from Light to Dark. All token-bound properties update instantly.

warning

Variable modes only work if the property is bound to a variable. If you have overridden a property with a hardcoded value, mode switching will not affect it.

Customizing tokens

To adapt Chisel UI to your brand without breaking the component structure:

Step 1 — Open the Variables panel
In the Figma file, go to Edit → Variables. You will see the "Chisel UI Tokens" collection.

Step 2 — Create a new mode
Click the + next to the existing modes. Name your new mode (e.g. "Brand Light").

Step 3 — Override values
In your new mode column, click any token value to override it. Start with these core tokens:

  • color/accent/gold → your brand primary color
  • color/accent/label → your brand primary color
  • color/bg/primary → your background color
  • color/text/primary → your primary text color

Step 4 — Apply your mode
Select any frame and switch its variable mode to your new brand mode. All components update instantly.

Step 5 — Create a dark variant
Repeat steps 2–3 to create a "Brand Dark" mode.

warning

Do not change the token names — components are bound to token names, not values. Changing a name breaks the binding. Only change the values in each mode column.

Components

Button

Overview
Buttons trigger actions. They are the most used component in the system and come in three visual styles reflecting three levels of hierarchy: primary (most important), ghost (secondary), and gold (special/featured actions).

Variants

By style:

  • Primary — filled, high contrast. Use for the single most important action in a section.
  • Ghost — outlined, lower contrast. Use for secondary actions alongside a primary button.
  • Gold — gold gradient. Use sparingly for special actions: "Open in Figma", featured CTAs, premium actions.
  • Icon — square, icon only. Use for compact toolbars or repeated actions where labels would clutter.

By size:

  • Large (lg) — 48px height, 24px horizontal padding. Use in hero sections, major CTAs.
  • Default — 40px height, 20px horizontal padding. Use in most contexts.
  • Small (sm) — 32px height, 16px horizontal padding. Use in tight spaces, inline contexts, table rows.

By state:

Default, Hover, Active (pressed), Focus, Disabled, Loading

Usage guidelines

Do

Use one primary button per section maximum. Pair primary with ghost for a two-button CTA. Use gold only for the most prominent action on the page. Use the large size in hero sections, default size everywhere else. Always include a text label unless it is an icon button in a toolbar.

Don't

Don't use more than one gold button per page. Don't use primary buttons for destructive actions — use a red-tinted ghost instead. Don't use icon buttons without a tooltip or aria-label. Don't use the loading state as a permanent state — it should resolve.

Anatomy

The button is composed of: container frame (auto layout, horizontal), optional left icon instance, text label, optional right icon instance. The 3D surface effect is achieved with a two-stop gradient on the fill and an inner shadow on the container.

Input

Overview
Text inputs collect user information. Chisel UI's input uses a minimal underline style by default, communicating precision and restraint. A full-border variant is available for contexts requiring a more traditional form appearance.

Variants

By style:

  • Underline — bottom border only. Minimal, preferred style.
  • Outlined — full border on all sides. Use when inputs appear in cards or panels where the underline would feel disconnected.

By state:

Default, Hover, Focus, Filled, Error, Disabled

By size:

Default (40px height), Large (48px height)

Usage guidelines

Do

Always include a visible label above the input. Use helper text to explain format requirements. Show error messages inline below the input. Use the outlined variant when inputs appear inside a card. Group related inputs with consistent spacing (24px gap).

Don't

Don't use placeholder text as a substitute for a label. Don't show error state before the user has interacted with the field. Don't use the disabled state unless the field genuinely cannot be interacted with.

Badge

Overview
Badges are small labels used to communicate status, category, or metadata. They are always small, always uppercase, and always supplementary — never the primary carrier of information.

Variants

By style:

  • Default — neutral, muted. Use for generic tags and metadata.
  • Gold — brand accent. Use for premium or featured status.
  • Success — green. Use for positive status (active, completed, verified).
  • New — highlights newly added items.
  • Updated — highlights recently modified items.

Usage guidelines

Do

Keep badge labels to 1–2 words maximum. Use badges to add metadata, not to replace a heading. Use the dot variant when space is extremely tight. Limit badges per element to 2 maximum.

Don't

Don't use badges as buttons. Don't use red/danger styling for badges — this pattern is reserved for alerts. Don't write badge labels in sentence case — always uppercase.

Toggle

Overview

By size:

Usage guidelines

Do

Always pair a toggle with a clear label. Use a toggle only for settings that take immediate effect. Add a description below the label if the consequence isn't obvious. Place the label to the right of or above the toggle.

Don't

Don't use a toggle in place of a checkbox in a form that requires saving. Don't use a toggle for binary choices between two named options. Don't use the toggle's on-color for anything other than the active state.

Tabs

Overview
Tabs organize content into sections within the same context. They are used when content belongs together but needs to be separated to reduce cognitive load.

Variants

By style:

  • Underline — active tab indicated by a bottom border. Minimal, preferred.
  • Pill — active tab has a filled pill background. Use in sidebars or cards.
  • Segmented — all tabs enclosed in a container, active tab fills its slot. Use for binary or small-set choices.

By state:

Default, Active, Hover, Disabled

Card

Overview
Cards group related information and actions into a contained surface. They are one of the most flexible components in the system.

Variants

By style:

  • Elevated — stronger shadow, no visible border
  • Ghost — border only, no background fill
  • Featured — accent border-top, slightly stronger surface

By content layout:

  • Horizontal — image/media on left, content on right
  • Vertical — image/media on top, content below (default)
  • Content only — no image/media slot

Modal

Overview
Modals interrupt the current context to present critical information or require a decision. Because they block the rest of the interface, they should be used only when the action or information genuinely requires focused attention.

By size:

  • Large — 960px max width. Use for content-heavy modals like component previews.
  • Small — 480px max width. Use for simple confirmations or alerts.

By type:

  • Content — title, body content, optional footer with actions
  • Confirmation — title, short description, two actions (confirm + cancel)
  • Alert — title, description, icon, single action

Patterns

Form patterns

Login form

Components: Input (email) + Input (password) + Button (primary, lg) + Button (ghost, sm for "Forgot password")

Use a 400px max-width container, centered. Vertical stack with 24px gap between fields, 32px gap before the primary button.

Search bar

Components: Input (outlined, with icon-left slot) + Button (primary or icon)

Use a full-width layout. The input takes remaining space, the button is fixed width. Add a keyboard shortcut hint (⌘K) in the placeholder.

Settings row

Components: List (settings style) with Toggle as the right element

Stack multiple settings rows in a list. Group related settings under a section label (Orbitron, 10px, uppercase). Add a divider between groups.

Navigation patterns

Top navigation

Components: Nav logo + List (navigation style, horizontal) + Button group (ghost + primary)

Fixed position at top. Use backdrop-filter: blur(12px) for the frosted glass effect. Max-width 1200px centered inner content.

Sidebar navigation

Components: Logo + List (navigation style, vertical) + Badge for counts

Fixed left, 240px width. Group items under section labels. Active item has left border accent in --accent2. Collapse to icons-only on smaller screens.

Tab bar (mobile)

Components: Tabs (underline or pill style)

Fixed at bottom of screen on mobile. 3–5 items maximum. Icons with labels below. Active state uses --accent2.

Feedback patterns

Empty state

Components: Icon (48px, --text3) + Heading (Audiowide, 18px) + Description (DM Sans, 14px, --text2) + optional Button

Center all elements. Use a muted icon relevant to the context. Keep the heading under 5 words.

Loading state

Components: Skeleton cards (same dimensions as real cards) with animated shimmer

Use --surface2 for shimmer base color. Animate using a moving gradient from left to right, 1.5s loop.

Error state

Components: Icon (red, 24px) + Heading + Description + Button (retry action)

Use --color-danger for the icon. Keep the description specific about what went wrong and what the user can do.

Success confirmation

Components: Icon (success, 24px) + Heading + Description + optional Button

Use --color-success for the icon. Keep the heading positive and specific. Auto-dismiss after 4 seconds if it is a toast.

Content patterns

Card grid

2, 3, or 4 columns depending on screen width. All cards in a grid must be the same variant. Consistent gap (24px at desktop, 16px at mobile). If one card has an image, all cards should.

List view

Full-width rows with a left thumbnail, middle content, and right action. Consistent row height (64px with thumbnail, 48px without). Alternating row backgrounds are optional — use --surface and --bg for alternation.

Stats display

Horizontal row of 3–4 stat items with vertical dividers between them. Each stat: large number (Audiowide) above a short label (Orbitron, uppercase). Center-aligned. Max-width 800px.

Theming

Built-in themes

Chisel UI ships with two themes: Light and Dark.

Light theme defaults

  • Background: #F7F7F7
  • Surface: #FFFFFF
  • Text: #1A1A1A
  • Accent: #C8A96E

Dark theme defaults

  • Background: #0A0908
  • Surface: #151412
  • Text: #F2EFE8
  • Accent: #C8A96E

Both themes are defined as variable modes in the "Chisel UI Tokens" collection. Every component uses these tokens — switching the mode on a frame applies the correct theme to every component inside it instantly.

Step 2 — Open Variables

Click the + button next to the existing Light and Dark modes. Name it after your brand.

Select a full-page frame, switch to your new mode, and review. Fix any tokens that look wrong. Repeat until the theme feels consistent.

If you want to change the typefaces, go to the text styles panel and update the font-family for each style.

What to keep

When customizing the theme, keep these unchanged for best results:

  • Spacing scale — changing spacing breaks component proportions
  • Border radius scale — changing this inconsistently creates visual noise
  • Elevation values — these are calibrated for both light and dark mode
  • Motion tokens — consistent timing is what makes interactions feel considered

  • The variants you need

Reporting a bug

A bug in the context of a Figma library means: a component that does not behave as documented, a token that is not applied correctly, a variant that is missing, or a dark mode inconsistency.

To report: open a GitHub issue with:

  • Component name
  • What is expected
  • What is actually happening
  • Screenshot

FAQ

Can I use Chisel UI for commercial client work?

Yes. The license allows use in any personal or commercial project. You may not resell the Figma file itself or repackage it as a competing component library.

Does Chisel UI work on the Figma free plan?

All components work on the free plan. Variable Modes (for switching between light and dark mode) require a paid Figma plan. On the free plan, components open in their default light mode.

Can I use components without the token system?

Yes, but you lose the ability to switch themes with a single click and any token overrides you make will be hardcoded. You will also lose the connection to any future token updates. Recommended only if you are not using Figma Variables.

How do I update my file when Chisel UI releases a new version?

If you have Chisel UI enabled as a team library, Figma will notify you when updates are available. Click "Review updates" to see what changed and accept the updates. If you duplicated the file to your personal drafts, you will need to manually re-duplicate the new version and migrate your work.

Can I mix Chisel UI with other component libraries?

Yes. There is no technical conflict. Style conflicts may occur if another library uses the same variable names — rename one set to avoid collisions.

Is there a React or code version coming?

It is on the roadmap but not yet scheduled. Follow the GitHub repository for updates.

What if the component I need does not exist yet?

Use the Foundations — build your component using the spacing scale, color tokens, radius tokens, and type scale. It will feel native to the system. Submit a component request on GitHub so it can be added in a future release.

Can I detach components?

Yes, but with caution. Detaching breaks the connection to the library — the component will no longer receive updates and you cannot swap variants. Only detach if you need to make structural changes that cannot be done through component properties. Always detach a copy, not the original.

What fonts do I need to install?

Audiowide, Orbitron, and DM Sans — all free on Google Fonts. Install them locally before opening the Figma file, otherwise Figma will substitute a fallback font and the layouts will break.
